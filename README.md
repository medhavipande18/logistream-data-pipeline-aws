# 🚀 LogiStream Supply Chain Data Pipeline: S3 to Redshift Data Warehouse

## 💡 Project Concept: Data Lakehouse for Real-Time Supply Chain Visibility

This project constructs a **Hybrid Lambda/Kappa Architecture** using serverless AWS services to ingest high-volume transactional data alongside complex semi-structured geospatial information (GeoJSON). [cite_start]The primary goal is to transform this diverse data into an optimized **Snowflake Schema** hosted on Amazon Redshift Serverless, enabling proactive operational alerts (e.g., late delivery detection) and strategic business analysis[cite: 22, 53].

---

## ⚙️ Architecture and Service Components

The pipeline is entirely implemented in **AWS US East (Ohio / us-east-2)**, utilizing a unified Glue Data Catalog for schema management.

| Component | Role in Pipeline | Key Function |
| :--- | :--- | :--- |
| **Amazon S3** | Data Lake / Staging Layer | Stores raw CSVs, GeoJSON files, and processed WKT output. |
| **AWS Lambda (Python)** | Data Pre-processing | [cite_start]Flattens the nested **`routes.geojson`** coordinates into **Well-Known Text (WKT) strings**, a format required for geospatial features in Redshift and Tableau[cite: 84, 159]. |
| **AWS Glue Crawlers** | Cataloging / Schema Inference | [cite_start]Scans S3 sources (raw and processed) and registers the schemas (e.g., `rawdata`, `processed_routes`) into the **`logistream_db`**[cite: 85]. |
| **AWS Glue ETL (PySpark)** | Core Transformation Engine | [cite_start]Implements the **dimensional modeling logic**, generating surrogate keys and joining raw data with processed routes and dimensions[cite: 87, 1130]. |
| **Amazon Redshift Serverless** | Data Warehouse (DW) | [cite_start]High-performance, columnar storage for the final **Snowflake Schema**[cite: 87, 830]. |
| **Amazon Athena** | Verification Layer | [cite_start]Used to query the Glue Catalog tables immediately after crawling to validate schema integrity[cite: 86]. |

## 🐍 ETL Transformation Logic & Schema Mapping

The core transformation logic is defined in `etl_jobs/glue_master_etl.py` and strictly follows the DDL defined in `data_warehouse/ddl_snowflake_schema.sql`.

### 1. Dimensional Load Strategy

All dimension tables are loaded using a **full overwrite (`mode="overwrite"`)** strategy, with PySpark performing the following steps:

1.  **Extraction:** Reads `rawdata` and `processed_routes` from the Glue Catalog.
2.  **Key Generation:** Surrogate keys (e.g., `product_key`, `customer_key`) are implicitly generated by the **Redshift `IDENTITY(1,1)`** column during the initial load, while natural keys are used for Spark lookups.
3.  [cite_start]**Hierarchy:** **`dim_department`** is created first, and its `dept_key` is then joined and added to the `dim_category` dataframe before loading (Snowflake Structure)[cite: 830].

### 2. Fact Table Linking (`fact_supplychain_events`)

The central fact table is constructed by joining the original raw data to the newly created dimensions using calculated and natural keys:

| Foreign Key (FK) | Source Columns / Logic | Dimension Table |
| :--- | :--- | :--- |
| **`product_key`** | `rawdata.product_card_id` | `dim_product` |
| **`customer_key`** | `rawdata.order_customer_id` | `dim_customer` |
| **`status_key`** | Composite Key on (`shipping_mode`, `delivery_status`, `order_status`) | `dim_execution_status` |
| **`route_shape_key`** | Composite Key on (`latitude_src`, `longitude_src`, `latitude_dest`, `longitude_dest`) | `dim_route_shapes` |
| **`order_date_key`** | `rawdata.order_date_dateorders` (Converted to YYYYMMDD integer) | `dim_date` |
| **`shipping_date_key`** | `rawdata.shipping_date_dateorders` (Converted to YYYYMMDD integer) | `dim_date` |
| **`order_geo_key`** | Composite Key on (`order_city`, `order_state`, `order_country`) | `dim_geography` |

## 🔑 Execution Status and Final Verification

The pipeline successfully executed after resolving initial regional configuration issues (the job was launching in the wrong region for the Data Catalog).

* **Final Load:** The ETL job successfully loaded all dimension tables and the central `fact_supplychain_events` table into the Redshift Data Warehouse.
